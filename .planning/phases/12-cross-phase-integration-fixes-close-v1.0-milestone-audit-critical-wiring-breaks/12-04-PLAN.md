    **Fixed pattern (apply identically to BOTH files):**
    ```ts
    const existingEditsMap = new Map<string, Record<string, string>>(

    const rows: ResultRow[] = rawResults.map((r) => ({
      // ...other fields unchanged...
      editedData: r.edited_data

      // ...
    ```

    MERGE STRATEGY RATIONALE (extend the Phase 6 STATE.md decision):
    - Phase 6 STATE.md decision: "Merge editedData on results hydration: fresh API data overwrites data fields but editedData map is preserved within session."
    - Phase 12 EXTENDS this: when the backend ALSO has edited_data (PATCH-written by Phase 9 endpoint), backend keys WIN over Zustand localStorage keys.
    - Spread order `{ ...existingEditsMap.get(r.filename), ...r.edited_data }` means backend overwrites Zustand for any key present in both.
    - Keys ONLY in Zustand (in-session edits not yet flushed to backend, e.g., debounce hasn't fired) are preserved.
    - Keys ONLY in backend are added.
    - When `r.edited_data` is null/undefined (pre-Phase-9 batches or rows with no edits): fall back to Zustand-only, then `{}`. Exact backward compat.
